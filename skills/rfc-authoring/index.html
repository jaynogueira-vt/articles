RFC Authoring

A method for writing an engineering RFC that is an implementation contract, not a product brief wearing an RFC hat: a canonical 11-section structure, per-section depth anchors, an alternatives discipline, and a research-first rule for every infrastructure claim.

Get the skill (Markdown)

In one paragraph

The most common failure in design docs is not missing sections: it is sections with the right names and product-brief content, so engineering review turns into product debate and nobody can start coding. This skill holds two lines. First, a firm boundary: the product doc owns WHAT and WHY, the RFC owns HOW, and content that crosses the line gets moved, not duplicated. Second, depth anchors: for every section there is a concrete bar ("a CREATE TABLE with constraints and the migration tooling named", not "we'll need a table"), and a section that cannot reach the bar gets an explicit pending-research marker instead of filler prose. Around that core it fixes the structure (eleven canonical sections, synthesized from Google Design Docs, Phil Calcado's RFC process, and Oxide's RFDs), requires real alternatives with rejection reasons, and enforces research-first: verify infrastructure claims against the code before asserting them.

What it is

  • A canonical 11-section structure: Summary, Context, Goals & Non-Goals, Proposed Design (system context, data model, API, state machine, errors, compatibility, performance), Alternatives, Cross-Cutting Concerns, Rollout, Open Questions, Approvals, Revision History, Appendix.
  • A boundary table separating product-doc concerns (audience, domain language, policy, launch windows) from RFC concerns (identifiers, endpoints, event schemas, staged rollouts).
  • Per-section depth anchors: what "shallow, reject" and "real depth, target" look like for each section.
  • A research-first workflow: read the actual repos, confirm the producer side of every consumed event, check the domain glossary before naming anything new.
  • Rules for when an RFC is warranted at all, and when a ticket or an ADR is the right size instead.
  • Agent-agnostic: written for any skill-aware AI coding agent; adapt the publish steps to your team's wiki or docs repo.

What it does

  • Forces at least three named alternatives with explicit rejection reasons, "do nothing" always among them.
  • Requires state machines to name invariants and concurrency behavior, and rollout plans to name the flag, the gates, and the kill switch.
  • Maps every plain-language success signal to a concrete event, metric, or span in the observability section.
  • Marks unverifiable claims explicitly (pending research, or inferred with a verification target) so reviewers get a worklist instead of a guess.
  • Resolves open questions from the codebase where possible, with evidence and a revisit trigger, then renames the section to Design Decisions once nothing is actually open.

What it does not do

  • It does not duplicate the product doc: policy, scope, and metric targets stay where they belong.
  • It does not accept filler prose where a section cannot reach its depth anchor; the gap gets marked, not hidden.
  • It does not assert a framework, queue, event, or pattern it has not verified in the code.
  • It does not edit the companion product doc, even when it finds an error there; it flags the owner instead.
  • It does not park answerable questions in Open Questions or file tickets as a substitute for an hour of code reading.

When to reach for it

"Write an RFC", "design doc", "technical spec", or an approved product requirements doc whose next step is engineering design. Best fit: changes that cross services or teams, introduce persistent state, or hinge on at least one decision where the rejected alternatives matter. Wrong fit: single-file refactors, bug fixes, and single-team decisions with no external surface (use an ADR).

The discipline, in six rules

  1. Hold the boundary: the product doc owns WHAT and WHY; the RFC owns HOW. Content that crosses gets moved.
  2. Reach the depth anchor: every section either meets its concrete bar or carries an explicit pending marker.
  3. Research before prose: verify infrastructure claims in the code; tag the rest as inferred with a verification target.
  4. Real alternatives: three or more, named, described, explicitly rejected; "do nothing" included.
  5. Ship-safe by design: invariants on the state machine, a kill switch on the rollout, observability mapped to concrete signals.
  6. Close your own questions: resolve what the codebase can answer, with evidence and a revisit trigger; leave open only what is genuinely undecided.
